## 1. Бизнес-контекст и «боль», которую решает продукт

| Поле               | Значение                                                     |
| ------------------ | ------------------------------------------------------------ |
| Краткое название   | Платформа управления индивидуальным учебным планом студентов |
| Проект (практикум) | Проектный практикум 1                                        |
| Уровень сложности  | A                                                            |
| Тип работ          | Прикладной (практико-ориентированный)                        |

**Описание задачи:** web-сервис, который помогает студентам собрать **индивидуальный учебный план (ИУП)** без стресса, путаницы и бесконечных согласований.

**Целевая платформа и ИИ (по постановке):**

- База: **Odoo**
- Интеллект: **LLM-агенты** с использованием **MCP** (Model Context Protocol) коннекторов

**Требуемый результат (по постановке):** современная онлайн-платформа с простым интерфейсом — **конструктор ИУП**.

**Критерии оценки (ориентиры продукта):**

- студент собирает план за несколько минут;
- система **автоматически** предупреждает об ошибках и **перегрузке**;
- интерфейс быстрый и понятный без долгого обучения;
- большинство планов проходит согласование без исправлений.

**Связь с репозиторием:** ниже в разделе «Фактическое состояние» зафиксировано, какие из этих пунктов уже закрыты кодом, а что относится к **дорожной карте / будущим итерациям**.

## 2. Фактическое состояние репозитория (кратко)

| Область                                                                | Статус                                                                                                                      |
| ---------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **Odoo 17** модуль `ic_urfu_module`                                    | Реализован: модели, UI, права, workflow, генерация **DOCX**, демо-данные                                                    |
| **Конструктор ИУП** в смысле Odoo                                      | Есть: формы, семестры, дисциплины, согласование студент ↔ преподаватель                                                     |
| **LLM / MCP**                                                          | **В коде этого репозитория не подключены** (нет MCP-клиента, нет отдельного bridge-сервиса в дереве проекта)                |
| **Автоматические предупреждения об «ошибках и перегрузке» в стиле ИИ** | **Не реализованы**; есть **валидации** при отправке плана и перед генерацией документа (см. модели)                         |
| **Python-пакет `app/`**                                                | Минимальный задел: `pydantic-settings`, **structlog** (конфиг логирования для **будущих** сервисов вне Odoo, напр. FastAPI) |
| **Деплой**                                                             | Описан **Render.com** (`render.yaml`, `DEPLOY.md`, корневой `Dockerfile`)                                                   |

Итог: репозиторий — это **рабочий Odoo-модуль UrFU** с документооборотом ИУП и **инфраструктурой** Docker/Makefile; **LLM+MCP** из описания практикума — **целевая архитектура**, не текущий исполняемый код.

## 4. Модуль Odoo `ic_urfu_module`

### 4.1 Манифест

- Файл: `ic_urfu_module/__manifest__.py`
- **Версия Odoo:** 17 (`17.0.1.0.2`)
- **Зависимости аддона:** `base`, `web`, `mail`
- **Внешние Python-зависимости:** `python-docx` (для генерации DOCX)
- **Данные:** security, views, `data/demo_users.xml`
- **Demo:** предметы и планы в `demo/`

### 4.2 Модели (ORM)

| Техническое имя           | Назначение                                                                                                |
| ------------------------- | --------------------------------------------------------------------------------------------------------- |
| `ic.urfu.subject`         | Дисциплина: часы, ЗЕТ, форма аттестации, тип (обязательная / по выбору)                                   |
| `ic.urfu.semester`        | Семестр внутри плана: номер 1–8, учебный год, M2M на обязательные и elective дисциплины                   |
| `ic.urfu.plan`            | Индивидуальный план: данные студента, программа, семестры, workflow, бинарник сгенерированного DOCX       |
| `ic.urfu.config.settings` | Настройки в `res.config.settings` / `ir.config_parameter` (шрифт, дефолты для дисциплин, институт и т.д.) |

**Workflow плана (`ic.urfu.plan.state`):**

`draft` → `submitted` → `approved` / `rejected` → при успехе и генерации — `generated`.

Наследование `mail.thread` / `mail.activity.mixin`: уведомления и переписка по плану.

**Валидации (не LLM):**

- при отправке на проверку: есть хотя бы один семестр и хотя бы одна дисциплина в семестрах;
- перед генерацией DOCX: заполненность полей, семестры с дисциплинами, состояние `approved`;
- удаление: только для `draft` и `rejected`.

### 4.3 Генерация документа

- Код генератора: `ic_urfu_module/doc_generator/doc_generator.py` — сборка DOCX через **python-docx**, шаблон оформления UrFU (шапка, таблицы семестров и т.д.).
- Вызов из модели: `IndividualPlan.action_generate_document` — подготовка словаря данных `_prepare_document_data`, запись файла в поле `document_file`.

### 4.4 Безопасность

- Группы: студент (`group_ic_urfu_student`), преподаватель (`group_ic_urfu_teacher`).
- Record rules: студент видит только свои `ic.urfu.plan`; преподаватель — все планы (см. `security/ic_urfu_security.xml`).

### 4.5 Демо и учётные записи

- Логины/пароли синхронизируются с конфигом `ic_urfu_module/config/demo_credentials.py`; XML пользователей генерируется скриптом `scripts/generate_demo_users.py` (цель `make update-credentials`).
- Типичные демо-логины после `make init`: `student` / `student`, `teacher` / `teacher` (см. `README.md`, `docs/CREDENTIALS.md`).

## 5. Пакет `app/` (вне Odoo)

Назначение (из `app/__init__.py`): прикладной слой для **сервисов вне runtime Odoo**.

Содержимое на данный момент:

- `app/core/config.py` — `pydantic-settings`: `ENVIRONMENT`, `LOG_LEVEL`, `SERVICE_NAME`, опционально `.env`.
- `app/core/logging.py` — настройка **structlog** (JSON в production, цветной консольный вывод в dev), упоминание сценария вроде **FastAPI lifespan** — как ориентир для будущего HTTP-сервиса.

**Важно:** в репозитории **нет** `main.py` FastAPI, роутеров и т.п.; пакет — **заготовка** под отдельный процесс (например MCP bridge или API), если его добавят позже.

## 6. Инфраструктура и инструменты

### 6.1 `local-env/`

- **`docker-compose.yml`:** сервис `db` (PostgreSQL 17), сервис `odoo`.
- **Сборка образа Odoo для dev:** `build.context` = каталог `local-env/`, **`dockerfile: Dockerfile`** → используется файл **`local-env/Dockerfile`** (Odoo 17 + `python-docx`), а не корневой `Dockerfile`.
- **Порты:** Odoo `8069`, PostgreSQL `5432`.
- **Тома:** данные БД и Odoo filestore.
- **Монтирование:** исходники модуля с хоста: `../ic_urfu_module` → `/mnt/extra-addons/ic_urfu_module`, конфиг `./odoo.conf` → `/etc/odoo/odoo.conf`.

### 6.2 Корневой `Dockerfile` (деплой, например Render)

- База: `odoo:17.0`
- Установка `postgresql-client`, **`python-docx`**
- Копирование `ic_urfu_module` в образ: `/mnt/extra-addons/ic_urfu_module`
- `ENTRYPOINT` — `/entrypoint.sh` (в репозитории скрипт может отсутствовать; без него `docker build` из корня не пройдёт — проверьте наличие файла при подготовке CI/CD)

### 6.3 `Makefile`

Основные цели:

| Цель                                                  | Смысл                                                                                                |
| ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `make init`                                           | Регенерация demo users XML + `scripts/init.sh` (docker, БД, установка модуля, синхронизация паролей) |
| `make bootstrap`                                      | Полная очистка volumes + `init`                                                                      |
| `make start` / `stop` / `restart` / `logs` / `status` | Управление контейнерами                                                                              |
| `make upgrade-module`                                 | `odoo -u ic_urfu_module` в контейнере                                                                |
| `make test-generator`                                 | Локальный прогон генератора DOCX                                                                     |
| `make lint` / `lint-fix` / `lint-fmt`                 | Ruff через `uv run`                                                                                  |
| `make fix-passwords`                                  | Скрипт сброса паролей демо-пользователей                                                             |
| `make db-status`                                      | Снимок состояния БД/модуля                                                                           |

### 6.4 `scripts/`

- **`init.sh`:** поднимает compose, ждёт PostgreSQL и Odoo, создаёт БД при отсутствии, ставит `ic_urfu_module` с демо, перезапускает Odoo, синхронизирует пароли `student`/`teacher` через `odoo shell`.

### 6.5 `pyproject.toml`

- Python `>=3.12`
- Зависимости: `pydantic-settings`, `python-docx`, `structlog`
- Dev: `ruff`

Использование: **`uv`** (см. правила проекта — `uv run ruff check .` и т.д.).

## 10. Возможное направление развития (спекуляция, не как факт из репо)

Если целиться в постановку с MCP + LLM, типичная схема (как в правилах проекта в `.cursor/rules`):

- Odoo остаётся источником истины по планам и правам.
- Отдельный процесс (FastAPI или другой) реализует MCP tools и вызов LLM.
- Связь Odoo ↔ bridge: JSON-RPC / HTTP / очередь — **пока не закреплена в этом репозитории**.
